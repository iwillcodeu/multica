# Multica Inbox 邮件同步方案（邮箱账号）

> 状态：设计草案 · 不含实现代码  
> 目标：在现有 **邮箱登录** 前提下，将写入 Inbox 的通知 **同步发送一封邮件到该用户的注册邮箱**（`user.email`）。用户在钉钉等企业邮箱客户端中收信即可，**不单独对接钉钉账号或开放平台**。

---

## 1. 背景与目标

### 1.1 现状（Multica）

- **登录**：用户使用 **邮箱** 登录；账号与 `user` 表中的 `email`（唯一）一一对应。
- **Inbox**：`inbox_item` 表持久化；写入路径为 `server/cmd/server/notification_listeners.go`（issue/评论/提及等事件后 `CreateInboxItem`），成功后事件总线发布 `inbox:new`，前端 WebSocket 刷新。
- **收件人**：成员对应 `user` 表，**无额外「绑定邮箱」需求**——通知对象的邮箱即其登录邮箱。

### 1.2 目标

- 每当为某 **成员** 产生一条新的 Inbox 条目时，**同一摘要内容** 通过 **SMTP** 发到该用户的 **`user.email`**。
- **不替代** 应用内 Inbox，而是 **旁路同步**；可通过工作区或用户设置 **开关**（产品定稿）。
- **与钉钉的关系**：若组织使用钉钉内置/关联的企业邮箱，用户会在钉钉里收到这些邮件；实现上仍是 **标准发信到邮箱**，无需钉钉 `userid`、`unionid` 或扫码绑定。

### 1.3 非目标（首期）

- 不接钉钉开放平台、不做工作通知 / 机器人消息（若未来需要，可另起方案）。
- 不回写邮件已读到 Multica Inbox。

---

## 2. 收件人与身份（无 OAuth）

| 项目 | 说明 |
|------|------|
| 收件地址 | 固定为 **`user.email`**（与登录邮箱相同）。 |
| 映射 | `inbox_item.recipient_type == member` 时，`recipient_id` → 查 `user` 表取 `email`；**不需要** `user_oauth`、`user_identity` 或钉钉用户映射表。 |
| 可选增强 | 若将来允许用户改登录邮箱，以当时 `user.email` 为准；首期与现网「邮箱登录」一致即可。 |

---

## 3. 总体架构

### 3.1 原则

1. **在 Inbox 落库成功之后** 再触发发信；邮件慢则 **异步队列**，不阻塞 HTTP。
2. **幂等**：同一 `inbox_item.id` 只投递一封邮件（outbox / 投递表去重 + 重试安全）。
3. **可观测**：失败重试、死信、按工作区统计成功率。
4. **范围**：默认仅 **`recipient_type == member`**；Agent 收件人不发邮件（与现网「人类收通知」一致）。

### 3.2 逻辑架构（示意）

```
Issue/Comment 事件
    → notification_listeners（现有）
    → CreateInboxItem（现有）
    → Enqueue 邮件任务（member 且开关允许）
    → 解析 recipient_id → user.email
    → 渲染模板（标题、摘要、Issue 深链）
    → SMTP 发送
```

### 3.3 与现有代码的衔接点（概念）

- 在「已成功 `CreateInboxItem` 且 `recipient_type == member`」路径上 **入队邮件任务**，不修改 Inbox 语义。
- **避免** 在 HTTP handler 内同步 SMTP；统一 worker / 队列。

---

## 4. 开关与配置（建议）

1. **工作区级**：是否启用「Inbox 邮件同步」、SMTP 发信参数（密钥走环境变量或 Secret，不入库明文）。
2. **用户级（可选）**：是否接收邮件；若默认全开，可后续再加「退订链接」关闭。
3. **无需**：成员单独「确认接收邮箱」——与登录邮箱一致；若产品仍要防滥用，可用 **一次性验证邮件**（可选，P1）。

---

## 5. 邮件内容与模板

### 5.1 建议字段

- **Subject**：`[Multica] {inbox title 或 issue 标题}`（多工作区可加 workspace 名防混淆）。
- **Body**：通知类型（`type`）、严重度（`severity`）、摘要（`body` 或 issue 标题）、**Issue 深链**（`PUBLIC_APP_URL`）、可选 actor 名称。
- **Footer**：关闭同步或偏好设置链接（若实现用户级开关）。

### 5.2 频率与合并（可选）

- 通知密集时可做 **digest**（如 5 分钟合并一封）—— 独立产品决策与队列设计（P1）。

---

## 6. 技术实现：SMTP

- **发件方**：部署或租户在阿里云企业邮、腾讯企业邮等开通的 **SMTP 发信账号**；组织若用钉钉企业邮箱，仍按该邮箱服务商提供的 **SMTP** 配置即可。
- **注意**：SPF/DKIM/DMARC 由发信域名管理员配置，降低进垃圾箱概率。

**说明**：不提供「钉钉 API 发邮件」的并行通道时，本节即为唯一外发路径。

---

## 7. 数据与配置

### 7.1 建议新增持久化（概念）

- **工作区**：邮件同步开关、SMTP 连接信息引用（非明文密钥）。
- **投递记录**：`inbox_item_id`、`user_id` 或邮箱哈希、状态、错误、重试次数。

### 7.2 环境变量（示例）

- `INBOX_EMAIL_SYNC_ENABLED`（或工作区配置表）
- `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_PASSWORD`（或密钥引用）
- `PUBLIC_APP_URL`（深链）

---

## 8. 安全与合规

- **租户隔离**：SMTP 可按工作区配置；成员只收到 **本工作区** 产生的 Inbox 对应邮件。
- **最小必要**：敏感环境可只发标题 + 链接，正文不全量进邮件。
- **审计**：同步开关变更、发送失败率。

---

## 9. 失败处理与运维

- **重试**：网络/5xx 退避；无效地址进死信。
- **降级**：关闭同步时仅写 Inbox，不入队。

---

## 10. 实施阶段建议

| 阶段 | 内容 |
|------|------|
| **P0** | 工作区开关 + SMTP + 投递幂等表 + 模板 + `member` 仅 |
| **P1** | 用户级退订、digest、验证邮件（若需要） |
| **P2** | 监控看板、多工作区主题行优化 |

---

## 11. 风险与待决问题

1. **发信域名与 DNS**（SPF/DKIM）：需运维配合。  
2. **多工作区**：同一 `user.email` 在多个工作区时，主题或正文注明 **workspace 名称**。  
3. **邮件量**：高峰是否必须 digest（产品定）。

---

## 12. 与现有模块的对应关系

| 现有能力 | 在本方案中的角色 |
|----------|------------------|
| `notification_listeners` + `CreateInboxItem` | 成功后触发邮件入队 |
| `user.email` | **唯一收件人**（与登录邮箱一致） |
| 事件总线 `inbox:new` | 仍服务 Web；邮件走独立队列 |
